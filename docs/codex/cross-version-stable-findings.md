# Cross-Version Stable Findings: VS Code Codex 插件与本地 codex app-server

## 1. 文档目的

本文档只用于提炼 **跨版本相对稳定** 的技术机制结论，来源于多个版本的 VS Code Codex 插件分析结果对照。

本文档 **不是** 某一个具体版本的源码事实清单，也 **不是** 统一后的“单版本真相文档”。  
它只回答以下问题：

- 哪些机制在不同版本中都表现出较高稳定性
- 哪些边界可以视为跨版本成立的总体判断
- 哪些内容虽然在某一版本出现，但不应在本文中当作稳定事实

## 2. 使用边界

本文档只收录满足以下至少一条的内容：

1. 在多个版本分析结果中都明确出现
2. 即使具体对象名或局部实现变化，整体机制仍保持一致
3. 属于架构边界级结论，而不是某个版本的局部实现细节

本文档不收录：

- 仅某一版本出现的对象名、函数名、混淆符号名
- 仅某一版本中可见、但另一版本未证实的主流程细节
- 任何迁移方案、实现建议、架构设计
- 基于外部经验对未见源码内容的推断

## 3. 证据等级

本文档中的结论只使用以下标签：

- **跨版本稳定**：多个版本分析结果共同支持，且不依赖具体小版本实现细节
- **跨版本大体稳定**：总体机制一致，但局部入口、对象名或路径可能变化
- **待当前版本复核**：历史版本支持，但当前基线版本尚未再次核对
- **不纳入稳定结论**：只在单一版本出现，或版本差异过大，不适合写入本文

## 4. 跨版本稳定结论

### 4.1 总体架构边界

#### 结论 1：插件总体上可视为“VS Code 宿主层 + 本地 codex app-server + WebView 展示层”的多层结构
- 标签：**跨版本稳定**
- 说明：
  - VS Code 扩展宿主负责激活、命令注册、面板/视图管理、与本地进程通信
  - 本地 `codex app-server` 承担底层 agent 协议、生命周期、消息流与运行时交互
  - WebView 主要承担展示和用户交互，不是远端真实通信边界
- 不纳入本文的细节：
  - 具体对象名
  - 具体混淆后的类名/函数名
  - 某一版中特定 provider 的实现写法

#### 结论 2：真实通信边界不在 WebView，而在宿主与本地 app-server 之间
- 标签：**跨版本稳定**
- 说明：
  - WebView 不应被视为与远端 OpenAI/Responses/WebSocket 直接通信的主要边界
  - 宿主层通过本地 `codex app-server` 建立请求/通知/响应链路
  - 因此分析插件交互链路时，必须重点看 extension host 与 app-server 的连接，而不是只看 WebView
- 不纳入本文的细节：
  - 某一版 WebView 消息名
  - 某一版 panel/sidebar 特定转发逻辑

---

### 4.2 宿主层职责边界

#### 结论 3：VS Code 宿主层稳定承担以下几类职责
- 标签：**跨版本稳定**
- 职责：
  1. 扩展激活与命令注册
  2. WebView / Sidebar / Panel / Custom Editor 等宿主 UI 生命周期管理
  3. 本地 `codex app-server` 进程启动、握手、消息路由、退出感知
  4. 宿主侧请求构造与结果分发
  5. fatal error、异常状态、进程终止等宿主状态传播
- 说明：
  - 不同版本中具体对象划分可能调整，但这些职责分工本身具有稳定性

#### 结论 4：宿主层既是 UI 编排层，也是 app-server 的协议适配层
- 标签：**跨版本大体稳定**
- 说明：
  - 宿主不是单纯 UI 壳，也不是底层运行时本身
  - 它通常承担“把 VS Code 用户动作翻译为 app-server 请求，并把 app-server 的通知/请求/响应翻译回 UI”的中间层职责
- 版本相关说明：
  - 具体由哪个类、工厂或 provider 承担翻译职责，可能会随版本变化

---

### 4.3 本地进程与通信机制

#### 结论 5：插件会启动本地 `codex app-server` 进程，并通过本地进程通信与之交互
- 标签：**跨版本稳定**
- 说明：
  - `codex app-server` 是插件技术链路中的核心本地执行体
  - 宿主层会负责其启动、初始化、消息发送、异常处理与退出感知
- 不纳入本文的细节：
  - 某一版的 spawn 参数
  - 某一版的工作目录推导实现

#### 结论 6：宿主与 app-server 之间存在结构化请求/通知/响应语义，而不是随意文本交互
- 标签：**跨版本稳定**
- 说明：
  - 无论具体编码形式如何，链路中都存在明确的方法/参数/结果语义
  - 这使得该交互更接近协议化的本地 RPC，而不是普通命令行问答
- 备注：
  - 若某版本中可进一步观察到 JSON-RPC 风格封装，可在版本文档中详述，但本文只保留总体稳定判断

#### 结论 7：stdout 与 stderr 在宿主处理中承担不同角色
- 标签：**跨版本大体稳定**
- 说明：
  - stdout 侧通常承载结构化协议消息或主要交互流
  - stderr 侧通常更接近日志、错误文本、诊断信息
  - 宿主会将异常、退出、fatal 等状态进一步传播到 UI 或内部状态机
- 版本相关说明：
  - 具体日志消费和 fatal 构造路径，需以版本文档为准

---

### 4.4 协议层稳定结构

#### 结论 8：协议层至少稳定区分三类边界消息
- 标签：**跨版本稳定**
- 类别：
  1. **ClientRequest**：宿主主动发起的请求
  2. **ServerNotification**：服务端单向通知
  3. **ServerRequest**：服务端主动请求宿主参与交互
- 说明：
  - 这是理解插件交互机制的核心边界
  - 某些高层事件语义虽存在，但不应替代这三类消息边界

#### 结论 9：高层事件语义与底层协议边界不是同一层概念
- 标签：**跨版本稳定**
- 说明：
  - 类似事件流、增量消息、计划更新、diff 更新等，更适合作为“高层事件语义”理解
  - 但在分析真实链路时，应优先区分底层的 Request / Notification / Request-back-to-host 三类边界
- 不纳入本文的细节：
  - 某个具体事件名是否在某版本可见
  - 某版本是否做了事件聚合或 UI 适配

---

### 4.5 生命周期与主流程

#### 结论 10：主流程稳定围绕 initialize / thread / turn 等生命周期展开
- 标签：**跨版本大体稳定**
- 说明：
  - 插件与 app-server 的主要业务流并非无状态消息交换
  - 其核心流程围绕初始化、线程创建/恢复、turn 启动/推进/中断等生命周期组织
- 版本相关说明：
  - 具体请求名、旧接口兼容层、某些入口路径可能变化
  - 但“线程/回合式组织”本身是稳定机制

#### 结论 11：恢复与中断属于主流程能力，而不是纯附属功能
- 标签：**跨版本大体稳定**
- 说明：
  - 交互链路中通常存在 resume / interrupt 或同类语义
  - 它们说明宿主与 app-server 之间存在明确的状态推进与控制机制
- 备注：
  - 具体接口名和入口点需以当前版本分析文档为准

---

### 4.6 审批与用户交互

#### 结论 12：命令审批、补丁审批、用户输入请求属于底层交互机制的重要组成部分
- 标签：**跨版本稳定**
- 说明：
  - 某些交互不是宿主本地 UI 自行决定，而是由 app-server 主动请求宿主参与
  - 这类机制通常包括：
    - command approval
    - file change / patch approval
    - request user input
    - 某些 tool call 相关交互
- 结论意义：
  - 审批流不是纯展示层逻辑，而是协议驱动的交互机制

#### 结论 13：宿主在审批链路中承担“接收请求—触发用户交互—回传响应”的中介职责
- 标签：**跨版本稳定**
- 说明：
  - app-server 会发起某些需要宿主参与的请求
  - 宿主接住后，转化为 UI 交互，再将结果回传
  - 这构成一条独立于普通消息展示的关键支线
- 不纳入本文的细节：
  - 哪个 UI 组件负责展示审批
  - 哪个对象在某一版里承接响应

---

### 4.7 流式输出与增量事件

#### 结论 14：插件链路中存在稳定的流式或增量输出机制
- 标签：**跨版本稳定**
- 说明：
  - agent message delta、command output delta、diff/plan 更新等都体现出增量事件流特征
  - UI 并非在最后一次性拿到完整结果，而是在过程中逐步接收更新
- 备注：
  - 具体事件名、事件分层、是否直接来自底层通知或更高层事件映射，需看版本文档

#### 结论 15：diff / plan / reasoning / terminal output 等内容不应默认视为前端本地计算结果
- 标签：**跨版本大体稳定**
- 说明：
  - 这些内容通常与服务端通知、事件流或 app-server 输出紧密相关
  - 在没有版本级证据前，不应把它们当作纯 WebView 本地派生状态
- 版本相关说明：
  - 某些展示层聚合方式、缓存方式、二次转换方式可能变化

---

### 4.8 宿主特有能力与底层通用能力边界

#### 结论 16：VS Code commands / menus / panels / custom editor / workspace API 等属于宿主特有能力
- 标签：**跨版本稳定**
- 说明：
  - 这些能力依赖 VS Code 宿主环境
  - 它们不应被误认为底层 agent 或 app-server 本身具备的能力

#### 结论 17：初始化、线程/turn 生命周期、审批协议、流式事件语义、本地 agent 进程生命周期等属于底层通用能力候选
- 标签：**跨版本大体稳定**
- 说明：
  - 这些能力不依赖某个具体 WebView 组件或 VS Code 面板本身
  - 它们更接近底层 app-server / runtime 的通用机制
- 注意：
  - 本文只做机制边界判断，不延伸到迁移或复用建议

## 5. 不纳入稳定结论的内容

以下内容原则上 **不应直接写入跨版本稳定结论**，除非后续更多版本再次证明其稳定性：

1. 某一版本中的混淆对象名、局部类名、私有函数名
2. 某一版特定命令名到特定对象的精确调用路径
3. 某一版 panel / sidebar / owner 仲裁细节
4. 某一版中可见但另一版未证实的高层事件名
5. 某一版中的旧协议兼容接口是否仍在主流程工作
6. 某一版特定的 fatal error 构造字符串或本地 message 名
7. 某一版具体的 spawn 参数、环境变量组装实现、WSL 分支写法

这些内容应保留在各自版本文档中，而不是提升为跨版本稳定事实。

## 6. 后续使用方式

本文档适合作为以下工作的输入材料：

- 后续继续做新版本插件技术提取时，作为“稳定机制基线”
- 对比新版本是否改变了核心通信边界、生命周期组织或审批机制
- 在未来单独项目中，把“插件实现细节”和“底层稳定机制”区分开

本文档不适合作为以下工作的直接依据：

- 直接作为某个具体版本的精确实现说明
- 直接作为迁移设计文档
- 直接作为 API 设计说明
- 直接作为 UI 交互设计说明

## 7. 待持续复核项

以下内容虽然可能具有跨版本稳定趋势，但当前不应直接提升为稳定结论：

1. 某些高层 EventMsg 是否始终稳定存在且持续被宿主主流程消费
2. 旧接口与 v2 接口在不同版本中的保留程度与真实使用程度
3. owner WebView / panel 之间的状态仲裁机制是否长期稳定
4. fatal error 广播路径中的具体消息命名是否长期稳定
5. diff / plan / reasoning 在不同版本中是“底层通知直出”还是“宿主二次聚合后转发”

这些问题应在未来新版本分析时继续校验。

---

## 8. 简要结论

基于多个版本分析结果，可以较稳定地确认：

- VS Code Codex 插件不是以 WebView 为真实通信边界，而是以 **宿主层 <-> 本地 codex app-server** 为核心交互边界
- 插件的核心机制长期围绕 **本地进程生命周期、结构化协议消息、线程/turn 生命周期、审批交互、流式事件更新** 展开
- VS Code 宿主层主要承担 **UI 宿主集成 + 协议适配 + 进程编排 + 状态传播** 的职责
- 具体对象名、调用路径、部分事件名和兼容接口是否仍在主流程中使用，属于 **版本相关事实**，不应直接提升为跨版本稳定结论
