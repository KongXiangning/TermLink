# CODEX_PHASE5_PERMISSION_CONTEXT_PLAN

## 1. 背景与目标

本计划对应 [REQ-20260309-codex-capability-mvp](../product/requirements/REQ-20260309-codex-capability-mvp.md) 中新增的以下条款：

1. `5.7 权限模式与审批可见性`
2. `5.9 命令确认弹窗`
3. `5.10 背景信息窗口`

同时对应本轮需求补充记录 [CR-20260317-0936-codex-app-permission-mode-and-context-window](../changes/records/CR-20260317-0936-codex-app-permission-mode-and-context-window.md)。

本计划是 [CODEX_CAPABILITY_IMPLEMENTATION_PLAN.md](./CODEX_CAPABILITY_IMPLEMENTATION_PLAN.md) 的 Phase 5 专项实施包，目标是补完 app 主链路产品化能力，而不是新增底层协议。

本专项只覆盖 3 件事：

1. 权限预设模式产品化。
2. 命令确认从消息卡片升级为阻塞弹窗。
3. “IDE 背景信息”入口替换为绑定当前线程的背景信息窗口。

约束：

1. 继续复用既有 `approvalPolicy` / `sandboxMode` 配置链路。
2. 继续复用既有 `codex_server_request` / `codex_server_request_response` 审批响应链路。
3. 背景信息窗口继续属于客户端展示层能力，不新增新的 app-server 协议。

## 2. 现状与差距

### 2.1 现状

1. 当前已有 `approvalPolicy` 与 `sandboxMode` 配置链路，并可随线程启动下发到上游。
2. 当前已有 `codex_server_request_response` 审批响应链路，且前端已具备 approval card 基础状态机。
3. 当前已有“IDE 背景信息”按钮，但其语义仍偏向旧入口命名，且未冻结为当前线程绑定窗口。
4. Android / WebView 当前已具备共享对话页与交互状态基础，不需要重新定义底层边界。

### 2.2 差距

1. 首页还没有用户可理解且持续可见的权限模式表达。
2. 命令确认仍停留在消息卡片思路，未形成 app 主路径的阻塞弹窗体验。
3. 背景信息入口还没有明确绑定到当前 `threadId`，也没有冻结自动刷新与空态规则。
4. 总计划虽已补入 Phase 5 目标，但尚缺少可直接实施的专项拆分与验收矩阵。

## 3. 实施拆分

### Phase 5A：权限预设模式与首页持续可见状态

目标：把底层双字段配置收口为 app 用户可理解的权限预设模式，并确保首页持续可见。

冻结口径：

1. UI 产品层新增 `permissionPreset` 概念，但不新增新的持久化协议字段。
2. 首页和新任务主路径只展示 3 种权限显示态：
   - `默认权限`
   - `完全访问权限`
   - `自定义权限`
3. 高级设置页继续保留底层双字段：
   - `approvalPolicy`
   - `sandboxMode`
4. 默认映射固定为：
   - `默认权限 = on-request + workspace-write`
   - `完全访问权限 = never + danger-full-access`
5. 任何不属于上述两档映射的底层组合，一律派生为 `自定义权限`。
6. 首页必须始终可见当前权限显示态。
7. 线程新建、恢复、切换时，首页显示必须从当前生效配置重新派生，不能依赖旧 UI 缓存。
8. 若设置页修改了底层双字段，首页权限显示必须立即同步刷新。

最小交付：

1. 权限模式入口收口到 app 主路径。
2. 首页显示正确。
3. 与线程启动实际下发配置一致。

### Phase 5B：命令确认弹窗

目标：把命令审批升级为 app 主路径的阻塞确认弹窗，同时不改底层审批协议。

冻结口径：

1. 继续复用既有 `codex_server_request` / `codex_server_request_response`。
2. `item/commandExecution/requestApproval` 与 `execCommandApproval` 在 app 端统一进入“阻塞命令确认弹窗”通路。
3. 本期主目标只保证命令确认弹窗。
4. `item/fileChange/requestApproval` 与 `applyPatchApproval` 继续保留现有卡片态，除非实现时可无风险复用，但不作为本专项验收前提。
5. 弹窗必须展示：
   - 说明文案
   - 原始命令
   - 允许
   - 拒绝
   - “记住此前缀”入口占位或真实选项
6. 弹窗未处理前：
   - 当前任务维持 `waiting_approval`
   - 首页状态与任务摘要明确显示阻塞中
   - 不允许用户误以为命令已继续执行
7. 手机端允许采用底部浮层而非中间 modal，但仍定义为阻塞弹窗，且优先级高于消息流，无需滚动消息区即可完成决策。

最小交付：

1. 命令审批时出现阻塞弹窗。
2. 允许 / 拒绝都能正确回写既有响应链路。
3. 未决前保持阻塞态。

### Phase 5C：背景信息窗口替换与线程绑定

目标：把当前“IDE 背景信息”入口替换为 app 专用背景信息窗口，并与当前线程绑定。

冻结口径：

1. 现有“IDE 背景信息”按钮改名并改义为“背景信息”。
2. 背景信息窗口的数据主键是当前 `threadId`，不是全局 session，也不是单次 turn。
3. 新建线程、恢复线程、切换线程时：
   - 如果窗口已打开，内容自动刷新
   - 如果窗口未打开，入口状态要能反映该线程是否已有背景信息
4. 线程为空、未物化、读取失败时统一显示空态，不保留旧线程内容。
5. 背景信息继续作为客户端展示层能力，不新增新的底层 app-server 协议。
6. 若当前已有背景信息数据源不足，先以“窗口框架 + 当前线程绑定 + 空态/刷新机制”作为最小交付。
7. 若背景信息后续包含工作区、线程摘要、上下文压缩使用量等多个来源，按“当前线程可读到的数据”渐进展示，不等待一次性补齐全量字段。

最小交付：

1. 入口替换完成。
2. 线程绑定成立。
3. 自动刷新与空态正确。

## 5. 测试与实施建议

### 5.1 单元 / 前端逻辑

1. `permissionPreset` 与 `approvalPolicy/sandboxMode` 的双向派生。
2. 自定义组合降级为 `自定义权限`。
3. 线程切换时首页模式显示刷新。
4. 背景信息窗口的 `threadId` 绑定、空态与刷新逻辑。

### 5.2 交互 / UI

1. 命令审批到来时出现阻塞弹窗。
2. 允许 / 拒绝都能正确回写既有响应链路。
3. 弹窗未决前任务保持阻塞态。
4. 背景信息窗口入口替换后，旧“IDE 背景信息”语义不再暴露给用户。

### 5.3 端到端 / 真机

1. `默认权限` 下需要确认的命令可以触发弹窗。
2. `完全访问权限` 下首页模式显示正确，且线程以 `never + danger-full-access` 启动。
3. 小屏下弹窗无需滚动消息流即可操作。
4. 背景信息窗口在 Android WebView 中可稳定跟随当前线程。

## 6. 风险与回滚

1. 风险：`permissionPreset` 被误实现为新的持久化协议字段。
   - 控制：在实现和评审中明确 `permissionPreset` 仅为客户端派生态。
2. 风险：权限显示态与真实下发配置脱节。
   - 控制：所有显示都从当前生效的 `approvalPolicy` / `sandboxMode` 真值重新派生。
3. 风险：命令弹窗只替换视觉，不真正阻塞任务状态。
   - 控制：验收必须包含“未决前维持 `waiting_approval`”。
4. 风险：背景信息窗口沿用旧全局缓存，切线程后残留上一任务内容。
   - 控制：窗口数据必须绑定当前 `threadId`，线程为空时主动回到空态。

回滚策略：

1. 若权限预设模式实现不稳定，可先回退到设置页底层双字段入口，但不得保留错误的首页显示态。
2. 若命令弹窗影响审批主链路，可临时回退到现有 approval card，但必须保留审批响应正确性。
3. 若背景信息窗口线程绑定未稳定，可先保留新入口命名与空态框架，暂时关闭自动刷新，不得继续展示旧线程残留内容。
