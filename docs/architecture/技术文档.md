# Codex Desktop / VS Code / codex-ipc 技术架构整理

## 1. 总体结论

当前 Codex Desktop 与 VS Code 的近实时同步，不应简单理解为多个 app-server 各自独立执行同一个 turn，也不应简单理解为 VS Code 通过 remote-control 直接控制 Desktop app-server。

更准确的模型是：

```text
app-server runtime 层
        ↓
owner UI / webview 状态层
        ↓
本机 codex-ipc IPC 总线
        ↓
follower UI / webview 同步层
```

其中：

* `codex app-server` 负责真实 runtime 能力，包括 thread、turn、工具执行、approval request、JSON-RPC stream。
* `codex-ipc` 负责同机多个 UI surface 之间的 owner/follower 协调。
* Desktop、VS Code、其他 surface 可以同时存在多个 app-server。
* owner 不是整机唯一，而是按 thread / conversation / surface 维度判断。
* 判断某一轮 turn 的实际 owner，应以工具子进程挂在哪个 app-server 下为准，而不是以哪个 UI 正在显示为准。

## 2. 分层架构

### 2.1 Runtime 层：codex app-server

`codex app-server` 是 Codex 的运行时服务层，主要职责包括：

* 管理 thread / conversation 生命周期。
* 启动 turn。
* 执行工具调用。
* 拉起 shell、PowerShell、MCP、CUA node runtime 等子进程。
* 发起 command approval、file approval、permissions request 等请求。
* 通过 JSON-RPC 或 app-server client 向 UI 层输出 stream event。
* 接收 UI 层返回的 approval decision、user input、interrupt、steer 等控制指令。

典型进程形态：

```text
codex.exe app-server --analytics-default-enabled
codex.exe app-server --listen stdio://
codex-app-server.exe --remote-control --listen ws://127.0.0.1:39998
```

其中：

* Desktop app 会启动自己的 app-server。
* VS Code 扩展会启动自己的 app-server。
* 自建应用也可以启动 `codex-app-server.exe --remote-control`。
* CUA / node_repl / MCP 相关链路可能会再拉起嵌套的 `codex.exe app-server --listen stdio://`。

### 2.2 UI 宿主层：Desktop / VS Code host

UI 宿主层包括：

```text
Codex.exe
Code.exe
VS Code extension host
Desktop renderer / webview host
```

这一层同时连接两类对象：

```text
一边连接 codex app-server
一边连接 codex-ipc
```

它承担桥接职责：

```text
app-server event
        ↓
owner UI state / webview store
        ↓
thread-stream-state-changed
        ↓
sendBroadcast 到 codex-ipc
```

以及反向控制：

```text
follower 操作
        ↓
thread-follower-* IPC request
        ↓
owner UI / webview
        ↓
app-server approval response / user input / control command
```

因此，UI 宿主层是 `app-server` 与 `codex-ipc` 之间的关键桥接层。

### 2.3 IPC 层：codex-ipc

`codex-ipc` 是本机 UI surface 协调总线。

Windows 下表现为：

```text
\\.\pipe\codex-ipc
\Device\NamedPipe\codex-ipc
```

它的核心职责包括：

* surface 注册。
* owner/follower 协调。
* thread stream state 广播。
* read state / archived state / queued follow-ups 等状态同步。
* follower approval/input/interrupt/steer 等请求转发。
* client status 传播。

它不是 app-server transport，不负责真实工具执行。

### 2.4 Webview / Surface 层

每一个 Desktop 或 VS Code 中的 Codex UI 面板，本质上都可以看作一个 surface / webview。

扩展宿主中存在类似：

```text
ipcClientsByWebview
registerIpcClientForWebview
disposeIpcClientForWebview
postMessageToWebview
getThreadRole
```

这说明 IPC client 是按 webview 维度注册和管理的。

每个 webview 会参与 owner/follower 判定：

```text
getThreadRole(webview, conversationId) => owner | follower
```

只有当前 conversation 的 owner webview 会处理 follower 发来的控制请求。

## 3. 核心链路

### 3.1 Stream 同步链路

当 owner app-server 产生模型输出、工具事件、approval 状态变化时，链路大致为：

```text
Desktop app-server / VS Code app-server
        ↓
owner UI app-server client
        ↓
owner webview state mutation
        ↓
thread-stream-state-changed
        ↓
UI 宿主 getIpcClientForWebview(...)
        ↓
sendBroadcast("thread-stream-state-changed", payload)
        ↓
codex-ipc
        ↓
其他 surface 收到 ipc-broadcast
        ↓
follower webview 更新 UI
```

这解释了 Desktop 与 VS Code 中同一 thread 几乎同帧更新的现象。

重点在于：

```text
follower 看到的 stream state 不一定来自它自己的 app-server。
它可能来自 owner surface 通过 codex-ipc 广播的 UI 状态变化。
```

### 3.2 Approval 链路

当 app-server 请求命令执行确认时，链路大致为：

```text
app-server
        ↓
owner UI 显示 approval
        ↓
owner UI 广播 thread-stream-state-changed
        ↓
follower UI 同步显示 approval
```

如果用户在 follower 上点击确认或拒绝：

```text
follower webview
        ↓
thread-follower-command-approval-decision
        ↓
codex-ipc request
        ↓
所有候选 surface 检查 getThreadRole(conversationId)
        ↓
只有 owner surface 处理该请求
        ↓
owner webview 收到 thread-follower-command-approval-decision-request
        ↓
owner webview 回 app-server approval response
        ↓
app-server 继续执行
        ↓
owner UI 广播 resolved / stream state
        ↓
所有 follower 同步更新
```

所以 follower 不直接控制 app-server。
follower 的操作先进入 IPC，再由 owner 转交给 app-server。

### 3.3 User input / steer / interrupt 链路

同理，以下 follower 操作也走类似链路：

```text
thread-follower-start-turn
thread-follower-compact-thread
thread-follower-steer-turn
thread-follower-interrupt-turn
thread-follower-update-thread-settings
thread-follower-edit-last-user-turn
thread-follower-submit-user-input
thread-follower-set-queued-follow-ups-state
```

这些请求都会先判断：

```text
getThreadRole(webview, conversationId) === "owner"
```

只有 owner surface 会真正处理请求。

## 4. Owner / Follower 模型

### 4.1 Owner 不是整机唯一

同一台机器上可以同时存在多个 app-server：

```text
Desktop app-server
VS Code app-server
另一个 VS Code 窗口 app-server
自建 remote-control app-server
嵌套 stdio app-server
```

因此 owner 不能按整机判断，而应按以下维度判断：

```text
conversationId
thread
surface / webview
runtime session
```

### 4.2 判断实际 runtime owner 的方法

判断某一轮 turn 的实际 owner，应看新增工具子进程挂在哪个 app-server 下。

例如：

```text
子进程挂在 Desktop app-server 下
=> Desktop 是该 turn 的 runtime owner

子进程挂在 VS Code app-server 下
=> VS Code 是该 turn 的 runtime owner

子进程挂在自建 codex-app-server.exe 下
=> 自建 remote-control server 是该 turn 的 runtime owner
```

不应仅凭 UI 上哪个窗口显示了消息来判断 owner。

### 4.3 Idle app-server 与 follower surface

某个 VS Code app-server 没有工具子进程，并不代表 VS Code 没有显示能力。
它可能处于以下状态：

```text
idle app-server
另一个窗口实例
follower surface
历史 thread owner
未参与当前 turn 的 runtime
```

## 5. 当前验证结论

当前验证结果支持以下判断：

```text
本轮工具执行 owner 是 Desktop app-server 14512。
VS Code 如果同步显示该 turn，则不是因为 VS Code 自己执行了工具。
更合理的解释是：VS Code 作为 follower，通过 codex-ipc 接收 owner UI 的 thread stream state broadcast。
```

同时，`handle64.exe -a codex-ipc` 的结果显示：

```text
Codex.exe 持有 \Device\NamedPipe\codex-ipc
Code.exe 持有 \Device\NamedPipe\codex-ipc
```

但没有看到：

```text
codex.exe app-server
codex-app-server.exe
```

直接持有该 pipe。

这进一步说明：

```text
codex-ipc 是 UI 宿主层机制，不是 app-server 直接机制。
```

## 6. 技术特点

### 6.1 runtime 与 UI 同步解耦

`codex app-server` 只关注 runtime：

```text
thread
turn
tool
approval
stream
```

`codex-ipc` 只关注 UI surface 间同步：

```text
broadcast
request routing
owner/follower
webview message
client status
```

这使得真实执行与多端展示解耦。

### 6.2 多 surface 本机协同

同一台机器上，Desktop 和 VS Code 可以同时显示同一个 thread 状态。
它们不需要都直接承载 runtime。

一个 surface 作为 owner 运行 turn，其他 surface 作为 follower 接收状态广播。

### 6.3 owner-gated request routing

follower 发出的控制请求不会被任意 surface 执行，而是通过：

```text
getThreadRole(conversationId) === "owner"
```

做 gating。

这降低了多 surface 同时操作同一个 thread 时的冲突风险。

### 6.4 几乎同帧同步

因为 follower 接收的是本机 IPC 广播，而不是独立公网 stream，所以 Desktop 与 VS Code 的 UI 状态可以非常接近同帧。

实际链路更像：

```text
owner UI state mutation
        ↓
local IPC broadcast
        ↓
follower UI update
```

而不是：

```text
model service
        ↓
Desktop stream
model service
        ↓
VS Code stream
```

### 6.5 app-server 可并存

多个 app-server 可以同时存在：

```text
Desktop owner
VS Code owner
remote-control owner
嵌套 stdio app-server
```

这意味着系统是多 runtime session 并存模型，而不是单全局 runtime 模型。

### 6.6 follower 可低成本同步

follower 不需要完整运行工具链，也不需要启动相同 MCP / CUA / shell 子进程。
它只需要接收 owner 广播的 stream state，就能同步 UI 展示。

### 6.7 第三方接入存在可能

理论上，第三方应用可以作为 observer 接入 `codex-ipc`：

```text
Termlink-demo
        ↓
连接 \\.\pipe\codex-ipc
        ↓
实现 IpcClient 协议
        ↓
监听 thread-stream-state-changed
```

但这不是 app-server 接入，而是 UI IPC 层接入。

## 7. Termlink-demo 接入边界

### 7.1 不建议直接让 codex-app-server 接 codex-ipc

当前证据显示：

```text
codex-app-server.exe 不直接参与 codex-ipc
```

因此不建议设计成：

```text
codex-app-server.exe --remote-control
        ↓
codex-ipc
```

更合理的是：

```text
Termlink-demo Node/TS 宿主层
        ↓
codex-ipc
```

也就是由 Termlink-demo 的 UI/服务层实现 IpcClient 协议，而不是修改 app-server。

### 7.2 第一阶段只做 observer

建议第一阶段只做只读监听：

```text
连接 \\.\pipe\codex-ipc
监听 broadcast
记录 thread-stream-state-changed
对比 Desktop / VS Code 当前 conversationId
```

不要一开始就发送：

```text
thread-follower-command-approval-decision
thread-follower-submit-user-input
thread-follower-interrupt-turn
thread-follower-steer-turn
```

这些属于控制类消息，容易扰乱 owner/follower 状态机。

### 7.3 第二阶段再做伪 follower

当 IpcClient 协议、握手、message framing、clientId、sourceClientId、request/response 格式都确认后，再考虑实现：

```text
thread-follower-submit-user-input
thread-follower-command-approval-decision
thread-follower-interrupt-turn
```

此时 Termlink-demo 才从 observer 变成真正 follower。

## 8. 推荐的最终架构描述

可以将整体架构概括为：

```text
Codex 是 runtime 与 UI 协同分层的架构。

app-server 负责真实运行：
- conversation / thread / turn
- tool execution
- approval request
- JSON-RPC stream

UI 宿主负责桥接：
- app-server client
- webview state
- postMessageToWebview
- IpcClient registration
- owner/follower routing

codex-ipc 负责本机多 surface 同步：
- broadcast thread state
- request routing
- follower 操作转交 owner
- 多窗口/多宿主协同

follower surface 不直接执行 turn。
它接收 owner surface 广播的 stream state；
它的 approval/input 操作经 codex-ipc 转交给 owner；
owner 再与 app-server 交互。
```

## 9. 一句话总结

`codex app-server` 是运行时，`codex-ipc` 是 UI 协同总线；二者之间不是直接连接，而是由 Desktop / VS Code 的 UI 宿主层桥接。owner surface 连接 app-server 并运行真实 turn，follower surface 通过 `codex-ipc` 同步状态并将用户操作转交给 owner。

## 10. 面向 Termlink-demo 新需求的核实结论

本节只回答“是否能实现”，不代表当前项目已经完成这些功能。

### 10.1 目标需求

目标是让 Termlink-demo 的前端 UI：

1. 只展示所选对话的信息；
2. 展示形式接近 Codex Desktop：
   - 用户消息
   - 进度更新
   - 最终回复
   - 状态摘要
3. 显示对话状态：
   - 进行中
   - 已完成
4. 支持交互：
   - 提权信息处理
   - PLAN 模式相关操作
   - 追求目标模式相关操作
5. 在本 UI 发送信息或执行操作后，另外两端也能同步。

### 10.2 结论总表

| 需求 | 是否可实现 | 结论级别 | 说明 |
| --- | --- | --- | --- |
| 只展示所选对话的信息 | 可以 | 已确认 | `thread-stream-state-changed` 自带 `conversationId`，当前项目已能按会话维护内存状态并可按会话过滤。 |
| Desktop 风格消息面 | 可以 | 已确认 | `conversationState.turns[].items[]` 中可区分 `userMessage`、`agentMessage(commentary/final_answer)`、`commandExecution`、`fileChange`、`mcpToolCall` 等，可在服务端先归约成用户可见 surface。 |
| 显示“进行中 / 已完成” | 可以 | 已确认 | 可由 turn 的 `status`、thread runtime 状态、以及现有 `ThreadState` 推导。 |
| 提权信息处理 | 可以 | 高可行，待 active 验证 | 协议层已识别 `thread-follower-command-approval-decision`、`thread-follower-file-approval-decision`、`thread-follower-permissions-request-approval-response`。 |
| PLAN 模式相关操作 | 可以 | 高可行，部分待 schema 抓取 | 展示层可直接从 `todo-list`、`agentMessage(commentary)` 还原；交互层至少可通过 `thread-follower-submit-user-input` 完成文本反馈。若涉及结构化选择，还需抓取 live payload。 |
| 追求目标模式相关操作 | 可以 | 高可行 | `/goal` 本质可作为普通用户输入进入 turn；若后续需要 UI 级 goal 管理，再补状态映射与专门控件。 |
| 本 UI 发消息后另外两端同步 | 可以 | 架构已确认，待 active 验证 | 这正是 owner/follower + `codex-ipc` 的职责；但当前项目此前刻意保持 observer-only，尚未对 live 控制消息做实际灰度验证。 |

### 10.3 为什么可以确认这些能力存在

#### A. 会话选择与消息面

`thread-stream-state-changed` 广播中包含：

- `conversationId`
- `change.revision`
- `change.conversationState`

而 `conversationState.turns[].items[]` 中已经观察到这些类型：

- `userMessage`
- `agentMessage`
  - `phase: commentary`
  - `phase: final_answer`
- `todo-list`
- `commandExecution`
- `mcpToolCall`
- `fileChange`
- `contextCompaction`

因此可以把底层 item 归约成接近 Desktop 的 surface：

- 保留：用户消息、commentary、final answer；
- 折叠：命令执行、文件变更、工具调用、上下文压缩；
- 过滤：reasoning 原始内容、patch 细节、内部协议字段。

#### B. 对话状态

当前代码与观测都说明状态可推导：

- turn 级 `status`
- `threadRuntimeStatus`
- 现有服务端 `ThreadState`

因此“进行中 / 已完成”不是协议缺失问题，而是 UI 映射问题。

#### C. 提权与 follower 控制

当前项目协议层已经列出了受保护控制方法：

- `thread-follower-submit-user-input`
- `thread-follower-command-approval-decision`
- `thread-follower-file-approval-decision`
- `thread-follower-permissions-request-approval-response`
- `thread-follower-interrupt-turn`
- `thread-follower-steer-turn`
- `thread-follower-start-turn`
- `thread-follower-compact-thread`
- `thread-follower-update-thread-settings`
- `thread-follower-edit-last-user-turn`
- `thread-follower-submit-mcp-server-elicitation-response`
- `thread-follower-set-queued-follow-ups-state`

这说明从架构上，Termlink-demo 不只是能观察，也能成为真正的 follower。

#### D. 两端同步

从前面已经确认的 owner/follower 机制看：

```text
Termlink-demo 作为 follower 发送控制消息
        ↓
codex-ipc owner/follower routing
        ↓
owner surface / owner host
        ↓
app-server
        ↓
新的 thread-stream-state-changed 广播
        ↓
Desktop / VS Code / Termlink-demo 同步刷新
```

所以“我在 Termlink-demo 发消息后，另外两端也同步”在架构上是成立的。

## 11. 实现要点记录（先记文档，不直接实现）

### 11.1 UI 数据面要分成两层

不要直接把 `thread-stream-state-changed` 的原始 patch 或完整 item 列表传给前端。

建议服务端先做两层模型：

1. `rawConversationState`
   - 仅服务端内存保存
   - 用于 patch 应用、revision 校验、构造后续控制消息

2. `desktopSurfaceSnapshot`
   - 发给前端
   - 只包含用户可见内容

推荐结构：

```ts
type DesktopSurfaceSnapshot = {
  conversationId: string;
  revision: number;
  status: "in_progress" | "completed" | "waiting_for_approval" | "failed" | "unknown";
  updatedAt: number;
  items: Array<
    | { kind: "message"; role: "user" | "assistant"; phase?: "commentary" | "final_answer"; text: string; turnId?: string; itemId?: string }
    | { kind: "status"; statusType: "commands" | "files" | "tool" | "context" | "approval" | "todo" | "goal"; text: string; turnId?: string }
  >;
  pendingApproval?: {
    kind: "command" | "file" | "permissions";
    requestId?: string;
    title?: string;
    description?: string;
    raw?: unknown;
  };
  pendingPlanAction?: {
    kind: "text_input" | "elicitation" | "queued_followups";
    raw?: unknown;
  };
};
```

### 11.2 只展示所选对话

前端必须有明确的 `activeConversationId`。

所有 UI 渲染都只基于：

```text
latest surface snapshot for activeConversationId
```

而不是：

```text
all recent ipc events mixed together
```

否则就会再次出现“别的对话的历史片段混入当前窗口”的问题。

### 11.3 对话状态映射

建议统一映射为：

| 源状态 | UI 状态 |
| --- | --- |
| `inProgress` / `running` | `进行中` |
| `completed` / `idle` | `已完成` |
| `waiting_for_approval` / request pending | `等待确认` |
| `failed` | `失败` |
| 其他 | `未知` |

状态来源优先级建议：

1. 当前 turn `status`
2. 已捕获的 pending approval / server request
3. thread runtime 状态字段
4. 兜底 `unknown`

### 11.4 提权信息的实现要点

要支持提权处理，至少要补三块：

1. **展示**
   - 从 surface 中识别“当前有待确认请求”
   - 渲染成确认/拒绝控件

2. **发送**
   - 命令提权：`thread-follower-command-approval-decision`
   - 文件提权：`thread-follower-file-approval-decision`
   - 权限请求：`thread-follower-permissions-request-approval-response`

3. **收敛**
   - 发送后不本地乐观写死结果
   - 以新的 `thread-stream-state-changed` 广播为准
   - 只有 owner 处理完成并广播后，Termlink-demo 才移除 pending UI

注意：当前项目尚未抓到足够多 live 提权 payload 样本，因此**具体 params schema 在实现前需要补抓包验证**。

### 11.5 PLAN 模式相关操作

这里要区分四类：

1. **展示型**
   - `todo-list`
   - `plan` / `Plan`
   - `<proposed_plan>...</proposed_plan>` agent message
   - commentary 中的计划说明
   - 流式生成过程中的 `item/plan/delta`

2. **普通文本反馈**
   - 使用 `thread-follower-submit-user-input`

3. **实施计划**
   - Desktop 生成 Plan 后，live request 是 `item/plan/requestImplementation`
   - request id 形如 `implement-plan:{turnId}`
   - params 中包含：
     ```text
     threadId
     turnId
     planContent
     ```
   - 点击“是，实施此计划”不能只发普通文本，否则 owner 仍可能处于 Plan Mode。
   - Desktop 原生行为等价于：
     ```text
     update-thread-settings-for-next-turn -> collaborationMode: default
     remove-plan-implementation-request
     send-follow-up-message -> "PLEASE IMPLEMENT THIS PLAN:\n{planContent}"
     ```
   - demo 侧可通过 IPC follower 链路近似实现：
     ```text
     thread-follower-update-thread-settings
             ↓
     thread-follower-start-turn
     ```
     其中 `turnStartParams.collaborationMode` 也必须带 default mode，避免本轮仍继承 Plan Mode。

4. **其他结构化交互**
   - 如果 PLAN 外还触发 elicitation / queued follow-ups，需要继续核实是否走：
     - `thread-follower-submit-mcp-server-elicitation-response`
     - `thread-follower-set-queued-follow-ups-state`
     - 或 `thread-follower-submit-user-input`

因此结论是：

```text
PLAN 原文展示可以做流式同步；
PLAN 实施按钮已确认走 item/plan/requestImplementation + follower start-turn；
普通调整意见仍可先按文本输入处理；
elicitation / queued follow-ups 仍需按具体 live schema 单独接入。
```

### 11.6 追求目标模式相关操作

“追求目标模式”至少有两层含义：

1. **把 `/goal ...` 作为用户输入发出去**
   - 直接可行
   - 本质是 `thread-follower-submit-user-input`

2. **把 goal 状态做成独立 UI 组件**
   - 从 surface 中归纳
   - 可把 goal 相关 commentary / final answer / todo-list 聚合展示

因此建议第一阶段只做：

```text
goal = 普通用户输入 + 专门的 surface 展示
```

不要一开始就为 goal 发明新的内部协议层抽象。

### 11.7 让另外两端同步的前提

要做到“本 UI 发消息，另外两端同步”，必须让 Termlink-demo 以 follower 身份发送真实控制消息，而不是只改本地 UI。

因此必须满足：

1. `CodexIpcClient` 允许 active send
2. 明确 `allowActive + confirmSend`
3. 每次控制消息带正确 method version
4. 使用当前 active conversation 的上下文
5. 以新的 owner 广播结果作为最终状态

当前项目里已有保护：

```text
control method 默认禁止发送
只有 allowActive && confirmSend 才放行
```

这是对的，后续实现 UI 时也应保留“显式开启 active follower 模式”的保护开关。

### 11.8 建议的实施顺序

建议不要一步到位，按下面顺序推进：

1. **P1: 单会话 surface UI**
   - 只展示所选 conversation
   - 显示用户消息、进度更新、最终回复、状态摘要、会话状态

2. **P2: 被动状态卡**
   - `等待确认`
   - `进行中`
   - `已完成`
   - `失败`

3. **P3: 提权控件**
   - 先做只读展示
   - 抓 live payload
   - 再接确认/拒绝发送

4. **P4: follower 发消息**
   - 先文本输入
   - 验证 Desktop / VS Code 同步

5. **P5: PLAN / goal 高级交互**
   - todo-list 展示
   - elicitation / queued follow-ups / goal 聚合

### 11.9 当前最重要的实现边界

可以确认：

- observer + 单会话 surface 展示，一定能做；
- follower 控制能力，协议层已经存在；
- 多端同步，架构上成立。

仍需补证据后再落地的点：

- 提权类消息的精确 params schema；
- PLAN 结构化交互的 payload 形态；
- queued follow-ups / elicitation 的 UI 触发条件；
- Termlink-demo 发出 active follower 消息后的 live 灰度验证。

当前核心服务端组件如下：

### 12.4 Stream state 聚合

`CodexIpcFeed` 只处理 `thread-stream-state-changed` 广播。
收到广播后会执行：

```text
ThreadStreamTracker.applyBroadcast(message)
        ↓
tracker.getConversationState(conversationId)
        ↓
buildDesktopSurfaceSnapshot(...)
```

`ThreadStreamTracker` 维护每个 conversation 的原始 state，并应用 IPC payload 中的 patch/revision 信息。
`buildDesktopSurfaceSnapshot` 再把原始 turns/items 转成前端可以直接渲染的轻量结构。

当前 snapshot 结构重点字段：

```ts
type DesktopSurfaceSnapshot = {
  conversationId?: string;
  revision?: number;
  status: "running" | "waiting_for_approval" | "waiting_for_input" | "completed" | "failed" | "interrupted" | "unknown";
  updatedAt: number;
  latestTurnId?: string;
  latestCollaborationMode?: Record<string, unknown>;
  latestDefaultCollaborationMode?: Record<string, unknown>;
  items: DesktopSurfaceEntry[];
  pendingApproval?: PendingApproval;
  pendingPlanAction?: {
    kind: "text_input" | "elicitation" | "queued_followups" | "plan_implementation";
    requestId?: string;
    requestMethod?: string;
    questionId?: string;
    acceptedAnswer?: string;
    turnId?: string;
    planContent?: string;
    canSubmit?: boolean;
    unavailableReason?: string;
    raw?: unknown;
  };
  pendingGoalAction?: { kind: "text_input" | "elicitation"; raw?: unknown };
};
```

当前可视化映射：

| 原始 item / state | demo 展示 |
| --- | --- |
| `userMessage` | 用户消息 |
| `agentMessage` + `commentary` | 进度更新 |
| `agentMessage` + `final_answer` | 最终回复 |
| `commandExecution` | 命令/工具状态摘要，或审批请求 |
| `fileChange` | 文件状态摘要 |
| `mcpToolCall` | 工具状态摘要 |
| `contextCompaction` | 上下文状态摘要 |
| `todo-list` | todo/plan/goal 相关展示 |
| `plan` / `Plan` | PLAN 原文，支持流式增量文本 |
| `<proposed_plan>` agent message | PLAN 原文兜底展示 |
| `state.requests[]` 中的 approval | 独立 approval request |
| `state.requests[]` 中的 `item/plan/requestImplementation` | 独立 PLAN 实施请求 |
| `reasoning` | 默认不展示 |

状态推断规则：

```text
pendingApproval 存在 -> waiting_for_approval
last turn status inProgress / in_progress -> running
pendingPlanAction / pendingGoalAction 存在且 turn 不在 running -> waiting_for_input
last turn status completed / done / finished -> completed
last turn status failed / error / cancelled -> failed
last turn status interrupted -> interrupted
最后 turn 有 items 但无 terminal status -> running
其他 -> unknown
```

### 12.5 WebSocket 推送协议

浏览器连接：

```text
ws://127.0.0.1:3002/ws
```

服务端推给浏览器的核心消息：

```text
codex_ipc_status
codex_ipc_sync_event
conversation_surface_snapshot
conversation_status_changed
conversation_action_required
follower_mode_changed
follower_message_sent
```

浏览器发给服务端的核心消息：


`set_active_conversation` 会做三件事：

1. 记录当前浏览器 client 的 active conversation
2. 订阅/focus 对应 thread
3. 如果 recent IPC events 中已有该 conversation 的 snapshot，立即补发最新 snapshot

后续 `CodexIpcFeed` 收到同一 conversation 的新 snapshot 时，`WebSocketGateway` 只推给当前选中该 conversation 的浏览器 client。

### 12.8 running 状态闸门

实测发现：如果在 Desktop 当前 turn 仍处于 `running` 时，demo 继续发送 `thread-follower-start-turn`，Desktop 会停留在“正在思考”，而 demo 可能已经从投影中看到后续输出，造成两边观感不同步。

因此当前实现加了双层保护：

1. 前端保护：`web/app.js`
   - `canStartFollowerTurn(conv)` 只允许以下状态发送：
     ```text
     completed
     interrupted
     failed
     unknown
     ```
   - `running` 时禁用输入框和按钮
   - 显示提示：
     ```text
     当前会话仍在进行中，暂不能从 demo 发起新任务
     ```

2. 服务端保护：`server/src/wsGateway.ts`
   - 发送前读取 latest conversation snapshot
   - 如果 status 不允许 start turn，直接返回：
     ```text
     Conversation is still running
     Wait for the current Desktop turn to finish before starting a new follower turn.
     ```

这个保护只针对“发起新 turn”。
后续如果要支持运行中的补充要求，应实现专门的 `thread-follower-steer-turn` 或相关 user-input/queued-followup 链路，而不是复用 `thread-follower-start-turn`。

### 12.9 PLAN 展示与实施链路

PLAN 当前分为两个阶段处理。

#### A. PLAN 原文流式展示

Desktop / VS Code 在生成 Plan 时，IPC state 中会出现小写 `plan` item，并通过以下事件持续补文本：

```text
item/plan/delta
```

owner webview 的行为也是把 delta 追加到 `type: "plan"` 的 item 上。因此 demo 的投影层必须同时识别：

```text
plan
Plan
<proposed_plan>...</proposed_plan>
```

当前实现位置：

```text
src/codex-ipc/thread-stream.ts
web/app.js
```

要点：

- `ThreadStreamTracker` 应用 IPC patch 后，`plan.text` 会随 revision 增长。
- `buildDesktopSurfaceSnapshot` 把 `plan` / `Plan` 转成 `kind: "plan_prompt"`。
- 生成中 turn 的状态优先保持 `running`。
- 只有 owner 广播 `item/plan/requestImplementation` 后，才生成可点击的 `pendingPlanAction`。
- Debug transcript 也追加 `item/plan/delta`，用于排查“主 UI 是否漏流”。

#### B. 点击“是，实施此计划”

live capture 已确认，Desktop 的 Plan 实施请求不是普通 `item/tool/requestUserInput`，而是：

```text
method: item/plan/requestImplementation
id: implement-plan:{turnId}
params:
  threadId
  turnId
  planContent
```

如果 demo 只发送普通文本“实施计划”，Desktop owner 仍可能处于 Plan Mode，最终报：

```text
当前仍处于 Plan Mode，我不能实际改写...
```

因此 demo 当前执行以下顺序：

```text
follower_plan_response
        ↓
WebSocketGateway.handleFollowerPlanResponse
        ↓
thread-follower-update-thread-settings
  threadSettings.collaborationMode = default
        ↓
thread-follower-start-turn
  turnStartParams.input = "PLEASE IMPLEMENT THIS PLAN:\n{planContent}"
  turnStartParams.collaborationMode = default
        ↓
owner Desktop / VS Code surface
        ↓
owner app-server 在 Default mode 下执行计划
```

`latestDefaultCollaborationMode` 从历史 snapshot / turn params 中提取，优先复用用户当前模型和 reasoning 设置。
如果没有历史 default mode，则用当前 collaboration mode 的 model / reasoning_effort 构造 default mode 兜底。

### 12.10 审批链路

审批显示来自两种来源：

1. turn items 中的 `commandExecution` pending approval
2. stream state 的 `requests[]`

投影层会生成：

```ts
pendingApproval: {
  kind: "command" | "file" | "permissions";
  requestId?: string;
  title?: string;
  description?: string;
  command?: string;
  availableDecisions?: string[];
  raw?: unknown;
}
```

前端点击按钮后发送：

```text
follower_approval_response
```

服务端当前已接入命令审批：

```text
thread-follower-command-approval-decision
```

关键修正点：

- UI 展示用的 `requestId` 不一定是 owner 需要的原始 ID
- 服务端会通过 latest snapshot 反查 `pendingApproval.raw.id`
- 如果存在 raw numeric/string id，优先用 raw id 发给 IPC
- decision shape 使用 Codex owner 期望的结构

这解决了“审批按钮能显示但点击无效”的问题。
