# TermLink Codex Stitch 2 Design Prompt

## 1. 使用方式

把下面 `Prompt` 整段复制给 Google Stitch 2。

如果你希望它先出 2 到 3 个方向，再收敛到一个高保真方案，可以先发送主提示词，再追加文末的“可选追加指令”。

## 2. Prompt

```text
你现在是一名资深移动端产品设计师和交互设计师。请为一个真实可工作的移动端 AI 终端产品设计高保真界面，不要把它设计成通用聊天 App，也不要做成夸张霓虹风 AI dashboard。

产品名称：TermLink

产品定位：
这是一个移动优先的 AI 终端控制台。用户在手机上连接远端会话，使用 Codex 协助完成开发和运维任务。它不是纯聊天工具，也不是纯终端模拟器，而是“AI 协作 + 终端上下文 + 任务线程管理 + 审批控制”合在一起的移动工作台。

设计目标：
请重设计 Android 容器内的 Codex 主工作区。这个页面运行在 Android WebView 中，上层已经有原生顶栏、会话抽屉、Settings、Workspace 等外围能力，但这次设计只聚焦 Codex 主工作区本身，不重做整个 App 框架。

核心要求：
1. 必须保留“终端仍然在场”的产品语义。Codex 不是替代 terminal，而是与 terminal 共存。当前产品结构是 terminal canvas + Codex panel 的混合工作区。
2. 设计必须是移动端优先，主要面向手机竖屏，同时兼顾大屏和横屏，但不要按桌面优先来设计。
3. 视觉方向走“专业控制台”，高信息密度、冷静、可信赖、克制，不要紫色发光主导，不要太像消费级聊天产品。
4. 默认中文界面文案语境。

请基于以下真实功能和操作逻辑进行设计，不要自行删掉关键能力，不要把高频核心能力藏得太深。

一、页面整体结构
- 这是 Codex 主工作区，不是单页聊天窗口。
- 页面中需要同时容纳：
  - 顶部状态区
  - 二级导航
  - 主日志/消息流
  - 运行态信息区
  - 计划模式工作流区
  - 输入区
  - 审批流弹层
- 需要明确处理移动端层级、滚动关系、键盘弹出后的布局变化、不同信息区之间的优先级。

二、顶部状态区必须表达的信息
- 当前线程状态：idle / running / streaming / waiting_approval / error
- 当前线程标题
- 当前工作目录 cwd
- 状态摘要
- 可见的中断操作

这个顶部区的目标不是做品牌横幅，而是做“任务控制条”。用户应一眼知道当前线程在做什么、是否卡住、是否需要审批、是否可以中断。

三、二级导航必须保留的入口
- 任务历史
- 运行态
- 工具

请重构成更适合手机窄屏的导航方式，但不要删掉这些入口。

四、任务历史页/面板需要支持的动作
- 打开当前线程或历史线程
- 创建分支
- 重命名线程
- 归档
- 恢复归档
- 显示状态标签：当前线程、已保存、已归档、处理中

任务历史不能只是一个简单列表，需要体现“线程是长期工作上下文”的感觉。

五、运行态不是一个普通日志框
运行态必须包含四类内容，并解决手机上的阅读负担：
- diff
- plan
- reasoning
- terminal output

请设计一种适合移动端的组织方式，让用户能快速切换或聚焦当前最重要的运行信息，同时不牺牲整体感知。

六、主日志区要求
- 这里承载 Codex 对话、系统反馈、执行进展
- 不能退化成普通 IM 聊天气泡体验
- 需要更像“工作流日志 + 对话协作”的结合体
- 应支持长内容、连续步骤、工具反馈、计划摘要等复杂信息块

七、输入区必须支持的能力
- 普通文本输入
- slash 命令菜单
- @ 文件提及
- 图像 URL 输入
- 单次模型覆盖
- 单次推理强度覆盖
- 单次沙盒设置覆盖
- 发送按钮

输入区是高频操作区，必须重点考虑：
- 单手可达性
- 键盘弹出后的可用性
- 长文本输入
- 输入附加项不能让界面过载

八、计划模式是一级能力，不是普通消息
用户有时会进入 plan mode，当前不会直接执行，而是先规划。

请为计划模式设计一个独立、可感知的工作流容器，需要能表达：
- 当前正在规划
- 计划摘要
- 继续补充
- 取消
- 后续执行

这个区域要让用户明显感知：现在处于“先讨论计划、再决定是否执行”的状态。

九、审批流必须是一级设计对象
不能只做 toast 或弱提示。需要设计完整的审批交互，覆盖：
- 命令确认
- 文件改动确认
- 补丁确认
- 补充信息请求

其中高风险命令审批必须是强阻断式 modal。用户应能快速理解：
- 为什么要审批
- 即将发生什么
- 风险点在哪
- 当前可执行的决策是什么

十、工具区至少包含
- 技能浏览
- 上下文压缩

此外页面还需要有“上下文窗口 / 上下文占用”的可视化入口，但不要让它比主任务更抢眼。

十一、设计时必须处理的真实使用场景
- 默认空闲态
- 运行中态
- 计划模式进行中
- 等待审批态
- 任务历史展开态
- 运行态展开态
- 输入聚焦与软键盘场景

请不要只输出单张首页，需要给出一套状态完整、可落地的移动端界面方案。

十二、体验要求
- 一眼看上去要像“移动端 AI 终端工作台”
- 必须有明确的信息层级，而不是所有信息平铺
- 对高频用户友好，适合连续长时间使用，不容易疲劳
- 避免误触
- 长内容滚动时仍保持定位感
- 关键控制项要稳定、清楚、易发现

十三、视觉语言要求
- 深色专业控制台风格
- 低饱和冷灰蓝系优先
- 强调结构、节奏、对比和信息扫描效率
- 可以有少量精致的高亮和状态色，但不要靠大面积炫光营造“AI 感”
- 字体、间距、卡片和分区要体现专业工具属性

十四、交付要求
请输出：
1. 整体设计概念说明
2. 信息架构说明
3. 关键界面高保真方案
4. 至少覆盖以下状态页：
   - 默认空闲态
   - 运行中态
   - 计划模式进行中
   - 等待审批态
   - 任务历史展开态
   - 运行态展开态
   - 输入聚焦与软键盘场景
5. 关键组件说明，包括：
   - 顶部状态区
   - 二级导航
   - 日志/消息区
   - 运行态卡片或切换结构
   - 计划模式容器
   - 输入区
   - 审批弹层
6. 说明每个设计决策如何服务“移动端 AI 终端工作台”这个定位

十五、验收标准
你的方案必须满足：
- 看起来不是聊天 app，而是 AI + terminal 的工作控制台
- 线程状态、计划流、审批流、运行态都被明确设计出来
- 手机上运行态四分区不会变得不可读
- 历史线程操作链路完整
- 审批设计具备足够强的阻断感和风险提示
- 键盘弹出后输入区依然可用
- 整体风格专业、可信赖，不是泛化霓虹 AI 风格

十六、边界条件
- 不要重定义后端协议
- 不要把外围的 Android 原生顶栏、Sessions 抽屉、Settings、Workspace 页面纳入本次重设计主体
- 只重构 Codex 主工作区的信息架构、交互结构和视觉语言

请先给出 1 个完整主方案。如果你认为有必要，可以在开头额外给出 2 个简短备选方向，但最终必须落到 1 套可直接进入实现讨论的高保真主方案。
```

## 3. 可选追加指令

如果你想让 Stitch 2 先做方向探索，再出主稿，可以在主提示词后追加这一段：

```text
先不要直接发散成很多张图。请先给我 2 到 3 个方向非常明确、差异足够大的移动端方案概念，每个方向用简短文字说明：
- 核心布局策略
- 信息优先级
- 为什么适合“移动端 AI 终端工作台”

然后选出你最推荐的 1 个方向，展开成高保真主方案。
```

如果你想让它更偏向实现可落地，而不是纯视觉稿，可以再追加这一段：

```text
请在输出中额外说明：
- 哪些区域适合固定
- 哪些区域适合折叠或切换
- 键盘弹出时哪些区域应保留、压缩、隐藏
- 哪些组件最适合优先实现为 MVP
```

## 4. 自检清单

在把 Stitch 2 的结果收回来以后，用这份清单快速判断是否跑偏：

1. 它有没有把界面做成普通聊天产品。
2. 它有没有把 terminal 语义完全做没。
3. 它有没有明确设计运行态四分区，而不是只留一个日志框。
4. 它有没有把计划模式单独设计出来。
5. 它有没有把审批流做成真正强阻断交互。
6. 它有没有覆盖手机键盘弹出场景。
7. 它有没有把任务历史当成真正的线程工作区，而不是普通侧边列表。
8. 它的色彩和氛围是否足够专业、克制、可信赖。
